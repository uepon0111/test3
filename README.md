# プロセカ リザルト管理

Project Sekai のリザルト画像を Google Drive に保存・一覧管理する GitHub Pages 用の静的Webアプリです。
画像は Tesseract.js (OCR) で自動解析し、曲名・難易度・楽曲レベル・判定内訳・コンボ数を読み取ります。

## 今回のアップデートについて

以下の改善を行いました。

- 画像をページに読み込む際・Google Driveにアップロードする際の進捗を、進捗バーとサイドバーの
  パーセント表示でリアルタイムに可視化しました。
- 矛盾(判定内訳とDBのノーツ数の不一致など)が残ったままアップロード/保存しようとすると、
  内容を一覧表示する確認モーダルを挟むようにしました。
- リザルトを手動で修正すると、その場で(数値入力はデバウンス後に)矛盾検知・レベルの自動補正が
  走るようにしました。編集専用で一度もOCRを実行していないレコードでも動作します。
- 曲名の入力欄(アップロード/編集フォーム、ページ内の曲名検索の両方)に、入力内容から
  リアルタイムで候補を表示するオートコンプリートを追加しました。**ひらがな入力にも対応**しており、
  読み方だけ分かっていて漢字表記が分からない曲名でも検索できます。
- Google Driveのアクセス範囲を、Drive全体から**このアプリが作成したファイルのみ**
  (`drive.file` スコープ)に縮小しました。あわせて、この縮小と相性が悪い旧命名法データの
  読み込み・作成サポートを廃止しました。
- 画像一覧から「編集・解析」を開いても画像が表示されず解析もできない不具合を修正しました。
- 読み取り範囲設定のラベルが範囲(クロップ枠)の中身と重なって見づらい問題を修正し、
  ドラッグ中は周辺を拡大表示するようにしました。
- 読み取り設定(機種プロファイル)をGoogleアカウントに紐づけてGoogle Driveに自動同期する
  ようにしました。設定を変更するたびに自動でDriveへ保存され、別の端末からログインしても
  同じ設定を利用できます。読み取り設定はログイン後にのみ表示されます。
- そのほか、選択モードの切り替え時にグリッドが再描画されない不具合など、見つかった問題を
  あわせて修正しています。

それぞれの詳細は以降の各セクションを参照してください。

## Google Drive 上のデータ構成とアクセス範囲

```
プロセカリザルト/            (ルートフォルダ)
  ├─ Results/               (リザルト画像を全てここにフラットに保存)
  │    ├─ MS29 曲名A (FC) 2026-07-08.png
  │    ├─ EX27 曲名B (FC-3) 2026-07-09.png
  │    └─ ...
  └─ settings.json           (読み取り設定=機種プロファイルの自動同期先)
```

曲ごとにサブフォルダを作らず、1つのフォルダに全リザルト画像を保存します。曲名・レベル・難易度・
判定内訳(PERFECT/GREAT/GOOD/BAD/MISS)・コンボ数などのデータは、各画像ファイルの
Drive カスタムプロパティ (`properties`) に保存しています。

### アクセス範囲 (スコープ) の縮小について

これまでは Drive 全体を読み書きできる `drive` スコープを要求していましたが、このアプリが
実際に必要とするのは自分で作った「プロセカリザルト」フォルダ配下だけのため、**このアプリが
作成したファイルにのみアクセスできる `drive.file` スコープ**に縮小しました。ログイン画面の
同意内容も「このアプリで使用したファイルのみ」という限定的な表示に変わります。

このアプリが過去に作成したフォルダ・ファイル(「プロセカリザルト」「Results」フォルダとその
中身)は、作成時のスコープに関わらずこの新しいスコープでも引き続きアクセスできます(Driveは
「どのアプリが作成したか」をファイル単位で記録しているため)。既存ユーザーは、スコープ変更に
伴い初回のみ再ログイン(再同意)が必要になる場合があります。

### 旧命名法サポートの廃止について

これまで運用していた `プロセカリザルト/FC/{レベル}{難易度} 曲名/FC` という構成(旧命名法)の
読み込み・書き込みサポートは廃止しました。スコープを縮小したことで、以前のような
「ファイル名に "FC" を含むものをDrive全体から検索する」広い検索を維持する意味が薄れたためです。
**旧命名法で保存された過去のデータは、このバージョンの一覧には表示されなくなります**
(画像自体はGoogle Drive上にそのまま残っているため、必要であればDriveアプリから直接確認できます)。

### 既知の不具合の修正について (旧バージョンをお使いだった方へ)

旧バージョンには、保存時に MISS 件数を読み違えて常に `0` として保存してしまう不具合が
ありました (`missDetail` というフィールド名のところを `miss` として参照していたのが原因)。
今回のバージョンで修正済みですが、**修正前に保存済みのレコードの MISS 件数・FC判定・
ファイル名内の表記はそのままでは直りません**。正しい値に直したい場合は、該当レコードを
「編集・解析」から選び、再解析または手動修正のうえ保存し直してください。なお、手動修正時は
今回追加したリアルタイム矛盾検知(後述)が働くため、総ノーツ数とコンボ数が食い違う場合など
おかしなデータに気づきやすくなっています。

また、一覧から「編集・解析」を開いても対象の画像が表示されず解析もできない不具合がありました。
一覧のサムネイル(`thumbnailLink`)は解像度が低いうえ、ブラウザのクロスオリジン制約により
canvas に描画してOCR解析することができない場合があったのが原因です。編集・再解析時は、
画像本体をDrive APIから認証付きで取得し直すよう修正しています。

## 画像の読み取り (OCR) について

リザルト画像から、以下の5項目を読み取ります。

- **難易度** (EASY/NORMAL/HARD/EXPERT/MASTER/APPEND)
- **楽曲レベル**。「楽曲Lv.APD34」のようなバッジから数値を読み取ります
- **曲名**
- **判定内訳** (PERFECT/GREAT/GOOD/BAD/MISS)
- **コンボ数**

### 項目ごとの二値化

読み取り範囲を切り出したあと、Tesseract に渡す前に二値化(白黒化)しています。バッジの配色
(難易度ごとに背景色が異なり、グラデーションのものもある)によって最適な処理が異なるため、
以下のように項目ごとに使い分けています。

- **難易度・楽曲レベル**: 色付き/グラデーション背景に白文字、という配色のため、「明度 − 彩度」
  という “白さ” の指標に対して大津の二値化(Otsu's method)を適用しています。単純な明度だけで
  判定すると、明るい色のバッジ(例: HARDの黄色)で白文字と背景が分離できなくなることがあるため。
- **曲名・判定内訳・コンボ数**: 単純な明度に対して大津の二値化を適用しています。
- どちらの方式でも「文字/背景のどちらが明側か」は、二値化後に画素数が少なくなる側
  (=面積が小さい側)を文字とみなして自動判定しています。

### 解析の進捗表示

画像の解析中は、モーダル下部に進捗バーを表示します。難易度・楽曲レベル・曲名・判定内訳・
コンボ数の5項目を読み取るたびに進捗が進み、複数枚をまとめて解析している場合は
「解析中... (2/5枚)」のように全体の進捗も表示されます。

### 総合判断 (result-reconciler.js)

曲名・難易度・楽曲レベル・総ノーツ数(PERFECT+GREAT+GOOD+BAD+MISSの合計)を突き合わせ、
自信の無い項目を他の情報から補います。

1. 曲名のOCR結果がマスターDBの楽曲と高い精度で一致すればそれを採用します。
2. 一致度が低い場合は、読み取った「難易度」と「総ノーツ数」から条件に合う楽曲をDBから逆算し、
   複数候補があれば曲名テキスト・楽曲レベルとの近さでさらに絞り込みます。それでも絞り込めない
   場合は自動確定せず、候補一覧を警告として提示します(誤った自動選択を避けるため)。
3. 曲名が特定できたら、その楽曲の全難易度の公式ノーツ数と読み取った総ノーツ数を突き合わせ、
   難易度の読み取りミス(色の見間違い等)も検出・自動修正します。
4. レベルはDB値を優先しつつ、OCR結果と食い違えば警告します。
5. 「コンボ数が総ノーツ数を超えている」「フルコンボのはずなのにコンボ数が総ノーツ数と
   一致しない」など、物理的にありえない組み合わせを検出して警告します。

これらの判断過程・警告は、画像ごとに保存され「実測ログ」から確認できます。

### 手動編集時のリアルタイム矛盾検知・解消

編集フォームで曲名・レベル・難易度・判定内訳・コンボ数を手動で修正すると、その場で
`result-reconciler.js` の判定が再実行され、フォーム上部の警告バナーとサイドバーの状態表示
(OK/確認/要確認)がリアルタイムに更新されます。数値入力は連続したキー入力ごとに走らないよう
少し(200ms程度)デバウンスしていますが、曲名オートコンプリートでの選択や難易度の切り替えは
即座に反映されます。

OCRの自動判断(上記)と異なり、手動編集中は**難易度を勝手には変更しません**(ユーザーが
難易度セレクトで明示的に選んだ値を、入力途中の判定内訳から自動で書き換えると混乱を招くため、
矛盾があれば警告のみ表示します)。一方で**レベルは自動補正します**。曲・難易度が決まれば
一意に決まる公式情報であり、ユーザーが任意に選ぶような値ではないためです。

編集画面を開いた直後(まだ何も編集していない状態)でも、保存済みのデータそのものに矛盾が
無いかを確認します。旧バージョンのMISS件数バグの影響が残ったレコードなどを見つけるのにも
役立ちます。

### アップロード/保存時の矛盾確認

「アップロード」「保存して反映」ボタンを押した時点で、未解決の警告・エラーが残っている
項目があると、その内容(曲名・矛盾の概要)を一覧表示する確認モーダルが表示されます。
「戻って確認する」で編集に戻るか、内容を把握したうえで「このまま続行」を選んで実行できます。

### 実測ログ

編集フォーム上部の「実測ログを見る」ボタンから、画像1枚ごとに以下を確認できます。

- 総合判断の結果(検出された矛盾・警告の一覧)
- 曲名の逆算などの判断過程
- 項目ごとの「切り出し範囲(座標)」「二値化後の画像」「OCRの生テキスト」「信頼度」

読み取り結果がおかしいときは、まずここでどの項目のどこに問題があったかを確認してください。
このボタンは、その画像を一度もOCR解析していない(手動編集のみの)場合は無効になります
(実測ログはOCR実行時にしか生成されないため)。矛盾の警告バナー自体は、OCR未実行でも
手動編集の内容から表示されます。

## 曲名の入力・検索 (オートコンプリート/かな対応)

以下の2箇所で、入力した文字からリアルタイムに候補が表示されるオートコンプリートを使えます。

- **アップロード/編集フォームの「曲名」欄**: 楽曲マスターDB([sekai-world/sekai-master-db-diff]
  の `musics.json`)の全曲が候補になります。候補を選ぶと曲名・マスターDBとの紐付け(レベルの
  自動入力・矛盾チェックの精度向上)が同時に行われます。
- **ページ内の「曲名検索」欄**: 現在読み込み済みのレコードに実際に存在する曲名のみが候補に
  なります(1件も無い曲を候補に出しても絞り込み結果が0件になるだけのため)。

どちらも**ひらがな入力に対応**しています。`musics.json` の `pronunciation`(読み方、ひらがな)を
使って照合しているため、正確な漢字・カタカナ表記が分からなくても、読み方(ひらがな)の一部を
入力するだけで候補が絞り込めます。カタカナで入力した場合もひらがなに変換して照合するため、
どちらの入力方法でも動作します(ローマ字入力そのものには対応していません)。

候補一覧は ↑/↓ キーで選択、Enterで確定、Escで閉じる、といった基本的なキーボード操作にも
対応しています。日本語IME入力中(変換確定前)の候補表示・Enterキーの扱いにも配慮しています。

## 読み取り範囲の設定

画面右上の「読み取り設定」ボタン(**ログイン後にのみ表示されます**)から、難易度・楽曲レベル・
曲名・判定内訳・コンボ数を読み取る範囲を、スクリーンショットの機種(解像度・アスペクト比)
ごとに保存できます。範囲はドラッグ&ドロップまたはピクセル単位の数値入力のどちらでも調整でき、
双方向に反映されます。

- 範囲のラベルは枠の外側(上)に表示されるようにし、小さい範囲でもクロップ枠の中身とラベル
  文字が重なって見づらくならないようにしています。また、編集中でない範囲のラベルは非表示にし、
  隣接する範囲同士でラベルが重なるのを防いでいます。
- ドラッグ・リサイズ中は、ポインタ付近に周辺を拡大表示する拡大鏡が出ます。指やカーソルで
  隠れがちな境界線を見ながら、画素単位で細かい位置合わせができます。

以前はこの設定画面が `settings.html` という別ページになっており、画面遷移のたびに
Googleログインのセッションが失われて「ログアウトされる」不具合がありましたが、
index.html 内のモーダルとして統合したことで解消しています(`settings.html` は旧リンク・
旧ブックマークからのアクセスをリダイレクトするだけの薄いファイルとして残しています)。

アップロード時は画像の解像度・アスペクト比から最も近い機種プロファイルが自動的に選ばれますが、
アップロード画面の「機種プロファイル」欄からいつでも手動で切り替えられます
(切り替えると、その場でその機種の読み取り範囲を使って再解析されます)。

### 設定のアカウント別Drive自動同期

機種プロファイルはブラウザの localStorage に保存されるのに加え、**ログインしているGoogle
アカウントに紐づけて Google Drive (`プロセカリザルト/settings.json`) にも自動的に同期**されます。

- ログインすると、まずそのアカウントのDrive上の設定を確認します。既にあればそれを読み込み、
  無ければ(このアカウントで初めて使う場合)ローカルの内容をそのままDriveへアップロードして
  以後の同期対象にします。
- 設定を変更する(プロファイルの保存・削除・インポート)たびに、自動でDriveへ書き込まれます。
  設定モーダル内に同期状態(同期中/同期済み/エラー)を表示します。
- 別の端末・ブラウザから同じGoogleアカウントでログインすれば、同じ読み取り設定がそのまま
  使えます。ブラウザをまたいだ手動でのエクスポート/インポートは、Driveに同期できない
  環境向けの手段として引き続き利用できます。

読み取り設定(ボタン・モーダルとも)はログイン後にのみ表示されます。設定自体がアカウントに
紐づく情報のため、ログインしていない状態では利用できません。

### デフォルトの読み取り範囲について

難易度・楽曲レベル・曲名・判定内訳・コンボ数のデフォルト範囲は、実際のリザルト画像を実測して
座標を決めています。ただしこれはあくまで一例の解像度・レイアウトに基づく初期値なので、
実際の見え方とズレる場合は設定画面から調整してください。特にコンボ数・判定内訳は機種によって
表示位置の個体差が出やすい項目です。

## アップロード/Driveへの送信について

- 画像をページに読み込む処理(サムネイル生成・機種プロファイルの自動選択)と、その後の
  OCR解析は、それぞれ進捗バーとサイドバーの状態表示でリアルタイムに確認できます。
- Google Driveへのアップロードは、送信バイト数から進捗を計算し(`XMLHttpRequest` の
  アップロード進捗イベントを使用)、画像ごとの送信率(例:「送信中 42%」)と全体の進捗バーの
  両方に反映されます。
- 既存レコードの「保存」はメタデータ更新のみ(画像本体は変わらないため再アップロードは
  発生しません)で、通常は一瞬で完了します。

### トランザクション(排他制御)について

Google Drive の REST API には、複数リクエストをまたいで原子的に処理する本来の意味での
「トランザクション」はありません。そのため、このアプリでは以下の方針で整合性を確保しています。

- **1ファイル単位の操作は単一リクエストで完結させる**: 新規アップロードは、画像本体と
  メタデータ(properties)を1回の multipart リクエストで同時に作成するため、
  「画像はあるのにデータが無い」ような中途半端な状態にはなりません。
- **フォルダ作成の排他制御**: 「プロセカリザルト」「Results」フォルダの検索・作成は、
  画像アップロードと設定のDrive同期の両方から呼ばれ得ます。これらがほぼ同時に走っても
  同名フォルダが重複作成されないよう、進行中の解決処理をPromiseとして共有し、後続の
  呼び出しは新たな検索・作成を行わずその完了を待つだけにしています。
- **設定同期の直列化**: 短時間に何度も設定を変更すると、書き込みリクエストがネットワーク上で
  追い越し合い、古い内容が後から返ってきて新しい内容を上書きしてしまう(最新の変更が
  失われる)おそれがあります。これを避けるため、Drive同期の書き込みは常に1件ずつ直列に実行し、
  実行中にさらに変更があった場合は完了を待ってから最新の状態で1回だけ追いかけて書き込みます。
