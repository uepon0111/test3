# プロセカ リザルト管理

Project Sekai のリザルト画像を Google Drive に保存・一覧管理する GitHub Pages 用の静的Webアプリです。
画像は Tesseract.js (OCR) で自動解析し、曲名・難易度・楽曲レベル・判定内訳・コンボ数を読み取ります。

  result-reconciler.js 曲名/難易度/レベル/総ノーツ数を突き合わせて最終値を決める「総合判断」・手動編集時の矛盾チェック

## Google Drive 上のデータ構成

```
プロセカリザルト/            (ルートフォルダ。このアプリが作成したフォルダのみを参照する)
  ├─ Results/               (リザルト画像はここにフラットに保存)
  │    ├─ MS29 曲名A (FC) 2026-07-08.png
  │    ├─ EX27 曲名B (FC-3) 2026-07-09.png
  │    └─ ...
  └─ settings.json           (機種プロファイル等の「設定」の同期用ファイル。後述)
```

曲ごとにサブフォルダを作らず、1つのフォルダに全リザルト画像を保存します。曲名・レベル・難易度・
判定内訳(PERFECT/GREAT/GOOD/BAD/MISS)・コンボ数などのデータは、各画像ファイルの
Drive カスタムプロパティ (`properties`) に保存しています。これにより、曲数が増えても
フォルダの作成・検索コストが増えず、一覧取得もフォルダ横断の突き合わせが不要になります。

過去に使われていた `プロセカリザルト/FC/{レベル}{難易度} 曲名/FC` という旧方式フォルダ構成の
読み取り・作成サポートは、このバージョンで廃止しました。旧方式のみで保存されていたデータは
一覧に表示されなくなります。必要な場合は、廃止前のバージョンで一度エクスポートするか、
手動で Results フォルダへ移してから再度読み込んでください。

### Google Drive アクセス範囲の縮小について

以前は Drive 内の**全ファイルを読み書きできる**権限 (`drive` スコープ) を要求していましたが、
このバージョンから**このアプリが作成したファイル・フォルダにしかアクセスできない**権限
(`drive.file` スコープ) に縮小しました。ログイン中に見えている「プロセカリザルト」フォルダの
中身以外には、そもそもアクセスする手段がなくなります。

この変更に伴い、**次回ログイン時に一度だけGoogleの権限確認画面が再表示されます**(権限の内容が
変わるため、Google側の仕様で再同意が必要です)。また、以前のバージョン(旧スコープ)で作成した
「プロセカリザルト」フォルダに対して、新スコープ下でも引き続きアクセスできるかどうかは
Google Drive側の内部的な判定に依存します。もし再ログイン後にこれまでのデータが一覧に
表示されない場合は、Google Drive を開いて「プロセカリザルト」フォルダの中身を新しく空になった
同名フォルダ(このアプリが新たに作成したもの)へ手動で移動してください。

## 今回のアップデートについて

運用中のフィードバックをもとに、主に以下を改善しました。

- **進捗の可視化**: 画像をページへ読み込む際(読み込み・OCR解析)と、Google Driveへ
  アップロードする際(バイト単位)の両方で、進捗バーを表示するようにしました。
- **矛盾検知の強化**:
  - Google Driveへのアップロード/保存ボタンを押した際、矛盾・警告が残っている項目があれば、
    内容を一覧表示して続行するかどうか確認するようにしました。
  - リザルトの各項目を手動で修正すると、その場でリアルタイムに矛盾を再チェックし、
    フォーム上部のバナーと画像一覧のステータスに反映するようにしました。
- **曲名のリアルタイム候補表示**: タイトルの手動登録欄と、ページ内の曲名検索欄の両方で、
  入力中の文字列からマスターDB([sekai-world/sekai-master-db-diff](https://sekai-world.github.io/sekai-master-db-diff/musics.json))
  を検索し、候補をドロップダウンで表示・選択できるようにしました。曲名(表記)だけでなく
  読み方(ひらがな)でも検索できるため、正式表記が分からない曲名でも読みで探せます。
  候補を選ぶと曲名・楽曲レベルが自動入力され、以後の矛盾チェックにも使われます。
- **画像一覧からの編集で画像が表示されない不具合を修正**: Driveのサムネイル用URLを
  そのままCORS付きで読み込もうとしていたことが原因で、編集モーダルを開いても画像が表示されず
  OCR解析もできない状態になっていました。認証済みAPI経由で画像本体を取得する方式に切り替え、
  解析に使う画像も(サムネイルではなく)元の解像度のものになるよう改善しています。
- **Google Driveのアクセス範囲を縮小**: 上記「Google Drive アクセス範囲の縮小について」を参照。
- **読み取り範囲エディタの見やすさ改善**: 範囲のラベルが枠の内側にあり、切り出し対象の文字と
  重なって見づらい問題を修正(ラベルを枠の外側上部に表示するよう変更)。また、ドラッグ操作中は
  周辺を拡大表示する拡大鏡を追加し、細かい位置合わせをしやすくしました。
- **設定のアカウント別保存・Google Drive同期**: 機種プロファイル(読み取り設定)をログイン中の
  Googleアカウントごとに分けて保存し、変更のたびに自動でGoogle Drive上の `settings.json` へ
  同期するようにしました。同じアカウントでログインすれば、別のブラウザ・端末でも同じ設定を
  引き継げます(同期状況は設定モーダル右上のバッジで確認できます)。
- **読み取り設定はログイン後のみ表示**: ログイン前は「読み取り設定」ボタン自体を非表示にしました。
- **旧命名方式(FCフォルダ)の読み取り・作成を廃止**: 詳細は上記データ構成の節を参照してください。
- **処理の耐障害性向上**: フォルダの検索・作成が複数箇所からほぼ同時に走っても重複フォルダが
  できないよう排他制御を追加したほか、一括削除は一部の項目が失敗しても残りを最後まで試行し、
  成功・失敗の件数を正確に報告するようにしました。
- **その他の修正**: 選択モード切り替え時に内部で未定義の関数を呼んでいたためエラーになっていた
  不具合の修正など。

### 追加の修正 (フィードバックを受けて)

- **設定のGoogle Drive同期が画面に反映されないことがある不具合を修正**: 同期処理自体は
  正しく動いていたものの、ログイン直後に読み取り設定を開くと、Driveからの取り込みが完了する前の
  ローカルの初期値が表示されたまま更新されないことがありました。ログイン処理の中でDriveとの
  同期完了を待ってから一覧取得へ進むようにしたほか、万一タイミングが重なった場合に備えて、
  同期完了時に設定モーダルの表示内容を自動的に最新化する仕組みを追加しました。
- **フルコンボ(FC)の判定条件を修正**: プロセカでは GOOD 判定でもコンボが途切れるため、
  本来は「GOOD・BAD・MISSが全て0件」の場合のみフルコンボと言えます。以前は「BAD・MISSが0件」
  だけを見てフルコンボ相当と判断していたため、GOODが発生しているのにコンボ数が総ノーツ数と
  一致しない(=当然の状態)場合にも誤って矛盾警告が表示される不具合がありました。判定条件に
  GOODが0件であることも含めるよう修正しています(OCR解析時・手動編集時の両方)。

## 画像の読み取り (OCR) について

リザルト画像から、以下の5項目を読み取ります。

- **難易度** (EASY/NORMAL/HARD/EXPERT/MASTER/APPEND)
- **楽曲レベル**。「楽曲Lv.APD34」のようなバッジから数値を読み取ります
- **曲名**
- **判定内訳** (PERFECT/GREAT/GOOD/BAD/MISS)
- **コンボ数**

### 項目ごとの二値化

読み取り範囲を切り出したあと、Tesseract に渡す前に二値化(白黒化)しています。バッジの配色
(難易度ごとに背景色が異なり、グラデーションのものもある)によって最適な処理が異なるため、
以下のように項目ごとに使い分けています。

- **難易度・楽曲レベル**: 色付き/グラデーション背景に白文字、という配色のため、「明度 − 彩度」
  という “白さ” の指標に対して大津の二値化(Otsu's method)を適用しています。単純な明度だけで
  判定すると、明るい色のバッジ(例: HARDの黄色)で白文字と背景が分離できなくなることがあるため。
- **曲名・判定内訳・コンボ数**: 単純な明度に対して大津の二値化を適用しています。
- どちらの方式でも「文字/背景のどちらが明側か」は、二値化後に画素数が少なくなる側
  (=面積が小さい側)を文字とみなして自動判定しています。

### 総合判断 (result-reconciler.js)

曲名・難易度・楽曲レベル・総ノーツ数(PERFECT+GREAT+GOOD+BAD+MISSの合計)を突き合わせ、
自信の無い項目を他の情報から補います。

1. 曲名のOCR結果がマスターDBの楽曲と高い精度で一致すればそれを採用します。
2. 一致度が低い場合は、読み取った「難易度」と「総ノーツ数」から条件に合う楽曲をDBから逆算し、
   複数候補があれば曲名テキスト・楽曲レベルとの近さでさらに絞り込みます。それでも絞り込めない
   場合は自動確定せず、候補一覧を警告として提示します(誤った自動選択を避けるため)。
3. 曲名が特定できたら、その楽曲の全難易度の公式ノーツ数と読み取った総ノーツ数を突き合わせ、
   難易度の読み取りミス(色の見間違い等)も検出・自動修正します。
4. レベルはDB値を優先しつつ、OCR結果と食い違えば警告します。
5. 「コンボ数が総ノーツ数を超えている」「フルコンボのはずなのにコンボ数が総ノーツ数と
   一致しない」など、物理的にありえない組み合わせを検出して警告します。

これらの判断過程・警告は、画像ごとに保存され「実測ログ」から確認できます。

**手動でリザルトを修正した場合**は、上記のOCR時点の判断とは別に、修正後の値同士に矛盾が
無いかをその場でリアルタイムに再チェックします(判定内訳の合計とDB上のノーツ数、コンボ数と
総ノーツ数の関係など)。OCRの判断のように自動で値を書き換えることはせず、あくまで警告の
提示に留めています(手動修正はユーザーの明示的な意思のため、値を勝手に上書きしないように
しています)。編集モードで既存レコードを開いた直後にも、保存済みデータに対してこのチェックが
一度走るため、以前保存した内容に矛盾が残っていないかもその場で確認できます。

### 実測ログ

編集フォーム上部の「実測ログを見る」ボタンから、画像1枚ごとに以下を確認できます。

- 総合判断の結果(検出された矛盾・警告の一覧)
- 曲名の逆算などの判断過程
- 項目ごとの「切り出し範囲(座標)」「二値化後の画像」「OCRの生テキスト」「信頼度」

読み取り結果がおかしいときは、まずここでどの項目のどこに問題があったかを確認してください
(このボタンはOCR解析を一度も行っていない項目では無効(押せない状態)になります)。

## 読み取り範囲の設定

画面右上の「読み取り設定」ボタン(ログイン後のみ表示されます)から、難易度・楽曲レベル・曲名・
判定内訳・コンボ数を読み取る範囲を、スクリーンショットの機種(解像度・アスペクト比)ごとに
保存できます。範囲はドラッグ&ドロップまたはピクセル単位の数値入力のどちらでも調整でき、
双方向に反映されます。ドラッグ中は周辺を拡大表示する拡大鏡が表示されるので、細かい位置合わせに
活用してください。

以前はこの設定画面が `settings.html` という別ページになっており、画面遷移のたびに
Googleログインのセッションが失われて「ログアウトされる」不具合がありましたが、
index.html 内のモーダルとして統合したことで解消しています(`settings.html` は旧リンク・
旧ブックマークからのアクセスをリダイレクトするだけの薄いファイルとして残しています)。

アップロード時は画像の解像度・アスペクト比から最も近い機種プロファイルが自動的に選ばれますが、
アップロード画面の「機種プロファイル」欄からいつでも手動で切り替えられます
(切り替えると、その場でその機種の読み取り範囲を使って再解析されます)。

機種プロファイルはログイン中のGoogleアカウントに紐付けて保存され、変更のたびに自動で
Google Drive上の `settings.json` へ同期されます。同じGoogleアカウントでログインすれば、
別のブラウザ・端末でも同じ設定を引き継げます。ブラウザの localStorage にもアカウントごとに
キャッシュを保持しているため、多少のオフライン時にも直近の設定が使えます。手動でのエクスポート/
インポート(JSON)も引き続き利用できます。

### デフォルトの読み取り範囲について

難易度・楽曲レベル・曲名・判定内訳・コンボ数のデフォルト範囲は、実際のリザルト画像を実測して
座標を決めています。ただしこれはあくまで一例の解像度・レイアウトに基づく初期値なので、
実際の見え方とズレる場合は設定画面から調整してください。特にコンボ数・判定内訳は機種によって
表示位置の個体差が出やすい項目です。

## 既知の不具合の修正について (旧バージョンをお使いだった方へ)

旧バージョンには、保存時に MISS 件数を読み違えて常に `0` として保存してしまう不具合が
ありました (`missDetail` というフィールド名のところを `miss` として参照していたのが原因)。
修正済みですが、**修正前に保存済みのレコードの MISS 件数・FC判定・ファイル名内の表記は
そのままでは直りません**。正しい値に直したい場合は、該当レコードを「編集・解析」から選び、
再解析または手動修正のうえ保存し直してください。
