# プロセカ リザルト管理

Project Sekai のリザルト画像を Google Drive に保存・一覧管理する GitHub Pages 用の静的Webアプリです。
画像は Tesseract.js (OCR) で自動解析し、曲名・難易度・楽曲レベル・判定内訳・コンボ数を読み取ります。

## ファイル構成

機能・役割ごとにファイルを分割しています。

```

  modals.css          画像モーダル・アップロード/編集モーダル・警告バナー等
  settings.css         読み取り設定 (範囲エディタ) 専用スタイル
  debug-log.css        実測ログモーダル専用スタイル

js/
  config.js           定数 (難易度定義・色・Driveフォルダ名・デフォルト読み取り範囲など)
  utils.js            汎用ヘルパー (文字列正規化・レーベンシュタイン距離・日付整形など)
  state.js             index.html 内で共有するグローバル状態
  device-profiles.js  機種プロファイルのCRUD (localStorage保存・自動選択・エクスポート/インポート)
  music-db.js         楽曲マスターDB取得・曲名のファジーマッチング・逆算候補検索
  result-reconciler.js 曲名/難易度/レベル/総ノーツ数を突き合わせて最終値を決める「総合判断」
  drive-client.js     Google Drive API の低レベル汎用処理 (検索・作成・アップロード・更新・削除)
  drive-records.js    リザルトデータ⇔Driveファイルの変換・取得・保存 (新方式/旧方式)
  personal-best.js    自己ベストの判定・グルーピング処理
  ocr-analyzer.js     Tesseract.js によるOCR解析 (難易度・楽曲レベル・曲名・判定内訳・コンボ数)
  debug-log.js         実測ログモーダルの描画 (切り出し画像・OCR結果・信頼度・警告)
  auth.js             Googleログイン/ログアウト
```

## Google Drive 上のデータ構成

### 新方式 (このバージョンから)

```
プロセカリザルト/            (ルートフォルダ)
  └─ Results/               (新規アップロードは全てここにフラットに保存)
       ├─ MS29 曲名A (FC) 2026-07-08.png
       ├─ EX27 曲名B (FC-3) 2026-07-09.png
       └─ ...
```

曲ごとにサブフォルダを作らず、1つのフォルダに全リザルト画像を保存します。曲名・レベル・難易度・
判定内訳(PERFECT/GREAT/GOOD/BAD/MISS)・コンボ数などのデータは、各画像ファイルの
Drive カスタムプロパティ (`properties`) に保存しています。これにより、曲数が増えても
フォルダの作成・検索コストが増えず、一覧取得もフォルダ横断の突き合わせが不要になります。

### 旧方式 (読み取り専用の後方互換)

これまで運用していた `プロセカリザルト/FC/{レベル}{難易度} 曲名/FC` という構成のデータも、
引き続き一覧に表示されます。旧データを編集して保存すると、その時点で自動的に新方式
(Resultsフォルダ)へ移行されます(画像本体の再アップロードは不要です)。

### 既知の不具合の修正について (旧バージョンをお使いだった方へ)

旧バージョンには、保存時に MISS 件数を読み違えて常に `0` として保存してしまう不具合が
ありました (`missDetail` というフィールド名のところを `miss` として参照していたのが原因)。
今回のバージョンで修正済みですが、**修正前に保存済みのレコードの MISS 件数・FC判定・
ファイル名内の表記はそのままでは直りません**。正しい値に直したい場合は、該当レコードを
「編集・解析」から選び、再解析または手動修正のうえ保存し直してください。

## 画像の読み取り (OCR) について

リザルト画像から、以下の5項目を読み取ります。

- **難易度** (EASY/NORMAL/HARD/EXPERT/MASTER/APPEND)
- **楽曲レベル** ※このバージョンで新規対応。「楽曲Lv.APD34」のようなバッジから数値を読み取ります
- **曲名**
- **判定内訳** (PERFECT/GREAT/GOOD/BAD/MISS)
- **コンボ数**

### 項目ごとの二値化

読み取り範囲を切り出したあと、Tesseract に渡す前に二値化(白黒化)しています。バッジの配色
(難易度ごとに背景色が異なり、グラデーションのものもある)によって最適な処理が異なるため、
以下のように項目ごとに使い分けています。

- **難易度・楽曲レベル**: 色付き/グラデーション背景に白文字、という配色のため、「明度 − 彩度」
  という “白さ” の指標に対して大津の二値化(Otsu's method)を適用しています。単純な明度だけで
  判定すると、明るい色のバッジ(例: HARDの黄色)で白文字と背景が分離できなくなることがあるため。
- **曲名・判定内訳・コンボ数**: 単純な明度に対して大津の二値化を適用しています。
- どちらの方式でも「文字/背景のどちらが明側か」は、二値化後に画素数が少なくなる側
  (=面積が小さい側)を文字とみなして自動判定しています。

### 総合判断 (result-reconciler.js)

曲名・難易度・楽曲レベル・総ノーツ数(PERFECT+GREAT+GOOD+BAD+MISSの合計)を突き合わせ、
自信の無い項目を他の情報から補います。

1. 曲名のOCR結果がマスターDBの楽曲と高い精度で一致すればそれを採用します。
2. 一致度が低い場合は、読み取った「難易度」と「総ノーツ数」から条件に合う楽曲をDBから逆算し、
   複数候補があれば曲名テキスト・楽曲レベルとの近さでさらに絞り込みます。それでも絞り込めない
   場合は自動確定せず、候補一覧を警告として提示します(誤った自動選択を避けるため)。
3. 曲名が特定できたら、その楽曲の全難易度の公式ノーツ数と読み取った総ノーツ数を突き合わせ、
   難易度の読み取りミス(色の見間違い等)も検出・自動修正します。
4. レベルはDB値を優先しつつ、OCR結果と食い違えば警告します。
5. 「コンボ数が総ノーツ数を超えている」「フルコンボのはずなのにコンボ数が総ノーツ数と
   一致しない」など、物理的にありえない組み合わせを検出して警告します。

これらの判断過程・警告は、画像ごとに保存され「実測ログ」から確認できます。

### 実測ログ

編集フォーム上部の「実測ログを見る」ボタンから、画像1枚ごとに以下を確認できます。

- 総合判断の結果(検出された矛盾・警告の一覧)
- 曲名の逆算などの判断過程
- 項目ごとの「切り出し範囲(座標)」「二値化後の画像」「OCRの生テキスト」「信頼度」

読み取り結果がおかしいときは、まずここでどの項目のどこに問題があったかを確認してください。

## 読み取り範囲の設定

画面右上の「読み取り設定」ボタンから、難易度・楽曲レベル・曲名・判定内訳・コンボ数を読み取る
範囲を、スクリーンショットの機種(解像度・アスペクト比)ごとに保存できます。範囲はドラッグ&ドロップ
またはピクセル単位の数値入力のどちらでも調整でき、双方向に反映されます。

以前はこの設定画面が `settings.html` という別ページになっており、画面遷移のたびに
Googleログインのセッションが失われて「ログアウトされる」不具合がありましたが、
index.html 内のモーダルとして統合したことで解消しています(`settings.html` は旧リンク・
旧ブックマークからのアクセスをリダイレクトするだけの薄いファイルとして残しています)。

アップロード時は画像の解像度・アスペクト比から最も近い機種プロファイルが自動的に選ばれますが、
アップロード画面の「機種プロファイル」欄からいつでも手動で切り替えられます
(切り替えると、その場でその機種の読み取り範囲を使って再解析されます)。

機種プロファイルはブラウザの localStorage に保存されます(サーバーを持たない静的サイトのため)。
別のブラウザ・端末でも使う場合は、設定画面の「エクスポート/インポート」で設定ファイル(JSON)を
移行してください。

### デフォルトの読み取り範囲について

難易度・楽曲レベル・曲名・判定内訳・コンボ数のデフォルト範囲は、実際のリザルト画像を実測して
座標を決めています。ただしこれはあくまで一例の解像度・レイアウトに基づく初期値なので、
実際の見え方とズレる場合は設定画面から調整してください。特にコンボ数・判定内訳は機種によって
表示位置の個体差が出やすい項目です。
